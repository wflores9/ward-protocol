9-Step Claim Validation

Each step must pass independently. All data is read from the XRPL ledger at claim time.

1
NFT Existence & Taxon Verification
Policy NFT exists in claimant's wallet with taxon = 281 (WARD_POLICY_TAXON)

Scans claimant's on-chain NFT list (paginated via AccountNFTs). Returns _WRONG_TAXON sentinel if the NFT exists but has incorrect taxon — preventing taxon spoofing. Returns None if the NFT is absent or burned, blocking replay attacks against destroyed policies.

AV 2.1 — Forgery AV 2.2 — Replay AV 2.13 — Taxon Spoof ward/validator.py WARD_POLICY_TAXON = 281
2
Policy Expiry Check (On-Chain Time)
Policy expiry read from NFT URI metadata, compared against XRPL ledger close_time — never server clock

Decodes NFT URI (hex → UTF-8 JSON). Supports compact format {"w":"ward-v1","v":…,"c":…,"e":…} and legacy {"protocol":"ward/v1",…}. Expiry is compared against the validated ledger close_time fetched via Ledger(ledger_index="validated") — eliminating clock-manipulation attacks.

AV 2.5 — Clock Manipulation ward/validator.py · ward/primitives.py get_ledger_close_time()
3
Vault Address Binding
Vault address in NFT metadata must exactly match the reported defaulted_vault argument

Cross-checks metadata["vault_address"] (or compact metadata["v"]) against the caller-supplied defaulted_vault address. A mismatch is an immediate rejection at step 3, preventing one vault's policy being claimed against a different vault's default. All addresses are validated with XRPL base58check at the API boundary (step 0, pre-flight).

AV 2.10 — Address Injection ward/validator.py validate_xrpl_address()
4
On-Chain Default Flag Verification
LedgerEntry for loan_id must exist and carry LSF_LOAN_DEFAULT flag (0x00010000)

Fetches LedgerEntry(index=loan_id) directly from the XRPL ledger. A successful response is treated as proof of default (the defaulted ledger entry exists). Vault loss is read from TotalValueOutstanding (falling back to PrincipalOutstanding then Amount). Runs concurrently with steps 1 and pool-info fetch via asyncio.gather.

AV 2.4 — Signal Manipulation ward/validator.py LSF_LOAN_DEFAULT = 0x00010000
5
Vault Loss is Positive
TotalValueOutstanding from the defaulted loan must be > 0 drops

Guards against zero-loss or negative-loss edge cases that could create trivially approvable claims with zero economic impact. All amounts are enforced as non-negative integers via validate_drops(), which rejects floats (XRP unit confusion), booleans, negatives, and values exceeding 100 billion XRP in drops.

AV 2.14 — Drops Unit Confusion ward/validator.py · ward/primitives.py validate_drops(vault_loss)
6
Pool Coverage Breach Check
Pool account balance must exceed XRPL reserve; usable balance ≥ 0 drops

Fetches pool's AccountInfo and computes usable = balance − (base_reserve + owner_count × owner_reserve). If usable < 0, the pool is insolvent and the breach flag is set. Reserves: 2 XRP base + 0.2 XRP per owned object (XRPL mainnet values). Pool drainage attacks (AV 2.8) are detected here.

AV 2.8 — Pool Drainage AV 2.9 — Coverage Ratio Manip. ward/validator.py XRPL_BASE_RESERVE = 2 XRP
7
Replay / Double-Spend Protection
NFT still live on-chain (not burned) — verified by successful step 1 response

Ward Protocol burns the policy NFT during settlement (NFTokenBurn). Because step 1 confirmed the NFT exists right now on the ledger, a previously-settled (burned) claim cannot pass step 1, giving implicit replay protection without a separate claims registry. Policy NFTs are TF_BURNABLE but NOT TF_TRANSFERABLE, preventing sale of a policy after a default is detected.

AV 2.2 — Replay AV 2.3 — Policy Transfer ward/validator.py TF_BURNABLE only (0x1)
8
Claimant Holds the Policy NFT
NFT found in claimant_address wallet, not any other account

The AccountNFTs request in step 1 is scoped to claimant_address. Finding the correct NFT ID there proves both existence (step 1) and ownership (step 8) atomically — the claimant must be the current holder. Pagination (up to 400 NFTs per page) ensures correctness for wallets with large portfolios.

AV 2.3 — Policy Transfer ward/validator.py AccountNFTs(account=claimant)
9
Pool Solvency & Rate Limit
Pool has sufficient usable balance ≥ payout; coverage ratio ≥ 1.5×; rate limit not exceeded

Two sub-checks: (a) Rate limit — max 3 claim attempts per NFT token ID per 300-second window (thread-safe sliding window, enforced before the expensive solvency check); (b) Pool solvency — usable balance must be ≥ payout AND coverage ratio ≥ 1.5. Payout is min(vault_loss, policy_coverage), capping the payout at the policy face value.

AV 2.8 — Pool Drainage AV 2.12 — Rate Limit Bypass ward/validator.py · ward/primitives.py MIN_COVERAGE_RATIO = 1.5

Core Invariants

  • ward_signed = FalseWard never holds or touches signing keys. Institutions sign; XRPL settles.
  • Events are hints, ledger is truthAll state re-fetched via independent RPC after WebSocket events.
  • 3-ledger confirmationDefault signals require health_ratio < 1.5 on 3 consecutive ledger closes.
  • TF_BURNABLE onlyPolicy NFTs use flag 0x1; TF_TRANSFERABLE (0x8) deliberately absent.
  • No off-chain trustZero reliance on oracles, off-chain databases, or signed attestations.
