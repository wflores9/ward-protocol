Getting started

Goal Run tests locally, then run the live testnet simulation and view on-chain results.

Prerequisites

  • Windows: use py -3 (Python Launcher) instead of python.
  • Dependencies: see requirements.txt.

Run tests

py -3 -m pip install -r requirements.txt
py -3 -m pip install pytest pytest-asyncio pytest-cov
py -3 -m pytest test_ward.py -m "not integration"

Run testnet simulation (writes testnet_run.log)

# one-time: create funded testnet wallets
py -3 scripts\setup_testnet_wallets.py

# run the simulation (UTF-8 recommended on Windows)
py -3 -X utf8 testnet_sim.py

View execution results

  • Local log testnet_run.log
  • On-chain Testnet explorer: https://testnet.xrpl.org/transactions/<HASH>
  • API health /health and /network/status

9-step claim validation

Ward’s claim path is designed to be ledger-verifiable: no oracle, no human judgment. The reference API endpoint is POST /claims/file.

Step 1 — Policy NFT exists (and is a Ward policy)

Verify the claimant holds the policy NFT on-chain and that its taxon matches the Ward policy taxon.

Step 2 — Policy is unexpired (ledger time)

Expiry checks use XRPL ledger time (not the server clock) to prevent local time manipulation.

Step 3 — Depositor/vault binding matches policy metadata

The policy NFT’s metadata (URI encoding) must bind to the expected vault and claimant context.

Step 4 — Default is verifiable on-chain

The loan must have the on-chain default signal (e.g. lsfLoanDefault) and be confirmed across multiple closes. Note On XRPL Altnet, XLS-66 ledger objects may not exist; the simulation demonstrates this gap.

Step 5 — Collateral ratio breached

Default must be economically justified by a provable collateral ratio breach (as defined by the protocol).

Step 6 — No active escrow pending

Reject if an escrow settlement is already pending for the same policy/claim context.

Step 7 — KYC credential valid and unexpired

Verify the depositor has a valid credential anchored on-chain (no raw PII on-ledger).

Step 8 — Domain credential valid

Verify the institution/domain authorization for the vault/flow.

Step 9 — No duplicates / rate limit clear

Reject duplicates and enforce attempt limits to reduce abuse and noisy retries.

Implementation note: the API returns CLAIM_REJECTED (422) with a reason when validation fails, or returns an unsigned settlement transaction when all checks pass.

Security invariants

Core invariant ward_signed = False. Ward never holds keys, never signs, never submits.

  • No custody / no keys: institutions sign and submit with their own wallets.
  • Policy NFTs are non-transferable by design: minted without tfTransferable.
  • Escrow settlement uses PREIMAGE-SHA-256: only the claimant holding the preimage can finish the escrow.
  • Ledger-time expiry: all expiry logic uses XRPL ledger time.
  • Multi-ledger confirmation: default events are only considered verified after multiple closes.
  • Replay protection: burn policy NFT after settlement; a burned policy can’t be reused.
  • Input validation: addresses and identifiers are validated before transaction construction.

Full threat model and mitigations: see security_notes.md in the repo.

Testnet proof

The repo includes testnet_proof.md with 5 confirmed on-chain transactions demonstrating premium payment, policy NFT mint, escrow create/finish, and NFT burn (replay protection).

  • Payment premium paid to pool
  • NFTokenMint policy issued (taxon 281)
  • EscrowCreate conditional escrow created
  • EscrowFinish payout released with preimage
  • NFTokenBurn policy burned after payout

You can reproduce locally with py -3 -X utf8 testnet_sim.py and verify each hash on the XRPL Testnet Explorer.
